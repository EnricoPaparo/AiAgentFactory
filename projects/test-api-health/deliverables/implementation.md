## Implementation Architecture

### Single Endpoint Design

```
HTTP Client
    ↓
GET /health
    ↓
Express Router
    ↓
Handler: Generate timestamp + return JSON
    ↓
HTTP 200 + {status: "ok", timestamp: "ISO8601"}
```

### Port Configuration

```
process.env.PORT
    ↓
    ├─ If set: Use provided value (e.g., 8000, 9000)
    └─ If unset: Default to 3000
    ↓
server.listen(port)
```

### Test Flow

```
Jest Test Suite
    ↓
    ├─ Import app from src/index.js
    ├─ Use supertest to make HTTP requests
    ├─ Assert status code, JSON structure, timestamp format
    └─ Report results (pass/fail)
```

## Decisions & Trade-offs

### D1: Express over alternatives
- ✅ **Chosen**: Express
- **Alternatives**: Fastify, vanilla Node.js, Koa
- **Rationale**: Explicit requirement, industry standard, well-documented for MVP

### D2: Jest + supertest for testing
- ✅ **Chosen**: Jest + supertest
- **Alternatives**: Mocha + Chai, Vitest
- **Rationale**: Jest is Node.js standard, supertest integrates perfectly for HTTP testing

### D3: Slim Docker base image
- ✅ **Chosen**: `node:20-lts-slim`
- **Alternatives**: `node:20-lts`, `node:20-alpine`, distroless
- **Rationale**: Balance between size (~160MB) and compatibility

### D4: Dynamic timestamp generation
- ✅ **Chosen**: `new Date().toISOString()`
- **Alternatives**: Fixed timestamp, server startup time
- **Rationale**: Realistic health check, validates server is running fresh

---

## Known Limitations

1. **No request logging**: Server only logs startup. Requests are silent (out of MVP scope).

2. **No error handling middleware**: Unhandled errors fall through to Express defaults (acceptable for stateless service).

3. **No graceful shutdown**: SIGTERM not handled. Process exits immediately (fine for containerized, stateless service).

4. **No rate limiting**: No protection against request flood. Not needed for health endpoint.

5. **No API versioning**: Single endpoint doesn't require versioning.
